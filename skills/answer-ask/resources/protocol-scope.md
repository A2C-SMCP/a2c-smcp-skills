# Answer Ask — 协议管辖判定指南（protocol-scope）

> 供 `answer-ask` Step 3 与 `consolidate-issues` Step 4 使用：判定一个三仓问询/冲突"协议管不管"。本文件是该判定的**单一源**。

## 判定总原则

**凡是出现在协议报文里、或影响跨端互操作行为的，协议管；只存在于单个 SDK 进程内部的，协议不管。**

一句话测试：**「如果 python-sdk 与 rust-sdk 在这一点上不一致，跨端通信会不会出错、或对端/Server 能否观察到行为差异？」** 会 → 协议管辖；不会 → SDK 自治（但建议双端对齐并留痕）。

---

## 协议管辖（Protocol-governed）

| 类别 | 说明 | 示例 |
|------|------|------|
| **事件定义** | 事件名、方向（`client:*` / `server:*` / `notify:*`）、触发时机 | 新增 `notify:*` 事件；`client:call_tool` 的触发条件 |
| **报文数据结构** | payload TypedDict 字段名/类型/可选性（`data-structures.md` 为 source of truth） | `CallToolRequest` 是否新增 `timeout` 字段；`tool_name` vs `toolName` 命名分歧 |
| **出现在报文中的默认值** | 会被序列化传输、或对端依赖的缺省语义 | 未传 `timeout` 时的默认超时语义；默认 Room 命名规则 |
| **错误码** | 错误码取值与语义 | 工具调用失败返回哪个错误码、含义是什么 |
| **Room / 会话机制** | 命名规则、隔离策略、广播范围 | Agent 与 Computer 是否共享 Room |
| **角色行为规范** | Agent / Server / Computer 职责边界与时序 | Computer 上线后是否必须主动上报工具列表 |
| **版本与兼容** | 版本号语义、升级路径、破坏性变更规则 | rc 版本是否允许破坏性变更 |

**示例 1（管）**：python-sdk 与 rust-sdk 对"未指定 room 时的默认 room 名"分别实现为 `"default"` 和 `"main"`。默认名体现在 Server 侧路由行为、跨端可感 → **协议管**。正确路径：升级协议裁决统一定义，双 SDK blocked-by 裁决结论跟进，禁止两侧各自继续开发。

**示例 2（管）**：rust-sdk 希望给 `CallToolRequest` 增加 `trace_id` 字段用于链路追踪。报文结构变更 → **协议管**，走 `/add-feature` 协议先行（评审 → 合并 → 发布 → SDK 跟进）。

## 协议不管辖（SDK 自治，建议对齐留痕）

| 类别 | 说明 | 示例 |
|------|------|------|
| **SDK 公开 API 形态** | 函数签名、类结构、构造器风格 | Python `Client(url=...)` vs Rust builder 模式 |
| **并发模型** | sync/async 双轨 vs Tokio | python-sdk 同步/异步双实现的组织方式 |
| **内部命名与组织** | 模块划分、内部变量名、目录结构 | 传输层模块叫 `transport` 还是 `io` |
| **日志与可观测** | 日志格式、级别、tracing 集成 | 是否默认输出重连日志 |
| **性能与资源** | 连接池、缓存、内部缓冲——前提是对端不可感知 | 内部缓冲区大小 |
| **测试 / 构建 / 发布** | 测试组织、CI、发布节奏 | pytest 分层 vs cargo features |
| **依赖选型** | 第三方库选择 | python-socketio vs rust_socketio |

**示例 3（不管）**：python-sdk 问 rust-sdk「客户端构造时是否应默认开启自动重连」。若重连与否不改变协议报文、Server 无感（仅本端连接生命周期管理），**协议不介入**。但这是双端开发者体验一致性问题 → 双 SDK 在发起问询的 Discussion 里直接对齐，结论以 Discussion 为准留痕，各自建实现 Issue 引用它。

**示例 4（不管 → 管的转化）**：重试策略看似 SDK 内部参数，但若重试会导致**重复发送带副作用的事件**（如重复 `client:call_tool`），就触及对端可感行为 → **升级为协议管辖**（需在协议中定义幂等/去重语义）。回答时要主动识别这类转化。

---

## 边界案例判定流程

1. 该差异是否改变**线上传输的内容**（事件名 / 字段 / 取值）？是 → 管
2. 对端或 Server 能否**观察到行为差异**？是 → 管
3. 仅本端进程内部差异？→ 不管；但影响双端一致性体验 → 建议对齐，结论留痕 Discussion，**不改协议文档**
4. 拿不准 → 按"管"处理，提交协议裁决——裁决结论即使是"协议不管"，该结论本身也是有价值的留痕，下次同类问题直接引用

## 协议身份回复的标准句式

- **管辖**：「本问题属**协议管辖**（<类别>）。当前规范定义见 <章节/结构>：…。如需变更，请走 `/add-feature` 协议先行流程，协议发布前双 SDK 不得按各自方案先行实现。」
- **不管辖**：「本问题属 **SDK 自治**范畴，协议不介入。建议 python-sdk 与 rust-sdk 就 <具体点> 在本 Discussion 直接对齐，结论以本 Discussion 为准留痕；后续若发现跨端可感差异，再升级协议裁决。」
