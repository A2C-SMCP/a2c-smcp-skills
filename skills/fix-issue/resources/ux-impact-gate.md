# 用户体验影响澄清门控（add-feature / fix-issue 共用）

> **单一源**。`fix-issue`（Step 3.6：协议门控后、写测试前）与 `add-feature`（Step 1.5：Issue 治理后、协议设计敲定前）各在自己的插入点引用本文件，不复制门控逻辑。与协议门控、隔离审查门控**同级正交**——都要过。

## 目的

A2C-SMCP 三个项目（rust-sdk / python-sdk / a2c-smcp-protocol）越来越偏**用户交互侧**，很多需求本质是「使用体验」的变更（典型如 marketplace plugin 的 install/enable 生命周期）。现有门控偏重**技术正确性**（协议合规、TDD、隔离审查），缺一道**体验正确性**门控——一旦变更触及用户可感知的交互，容易凭技术分析就把方案定了，而没先让用户从「使用体感」评估这到底是**进步还是退步、是否接受、此场景变强是否让别的场景变弱**。

**真实案例**：rust-sdk#102（plugin install/enable 分离）。技术上「协议先行 + config-first」分析得很干净，但真正的决策点是**产品语言**——「安装」该像装手机 App（装 ≠ 开）还是像装浏览器插件（装即开）。这只能靠用户的场景体感拍板，纯技术分析定不了。

## 触发条件

变更触及**用户可感知的交互 / 体验**即触发：

- UI 状态、操作步骤、命令行为
- 可见能力（skill / tool）出现或消失的时机
- 默认行为、生命周期语义（如「安装即启用」与否）

**豁免**：纯内核 / 协议内部、无用户可感差异的变更。豁免时须在计划中**显式说明已豁免**。

## 硬性规则

未经用户对「体验 before/after」做出判断并接受：

- **fix-issue**：不得进入写测试（Step 4）/ 动代码（Step 5-6）。
- **add-feature**：不得敲定协议设计（Step 2 / Step 3）。

与协议门控、cross-ask 门控、隔离审查门控**同级**。本门控在**动代码 / 定协议之前**，避免「技术方案已成型才发现用户觉得是退步」的返工。

## 强制产出模板（场景化、体感式，而非技术清单）

用 AskUserQuestion 把下述五段交用户拍板，**产出形态要具体到按钮点击、屏幕变化、用户内心 OS，而非协议条款堆叠**：

1. **舞台**：谁在用、在什么界面、有哪些操作入口。
2. **现状走查**：跟着用户操作走一遍，描述「屏幕上发生什么 + 用户心里什么感觉」，点出别扭 / 顺畅处。
3. **改后走查**：同样走一遍，描述改后的体感。
4. **跨场景权衡**：本场景变强，是否让**其它场景 / 其它用户群**变弱？显式列出**受益场景与可能受损场景**——不能只报喜。
5. **决策取景框**：把选择还原成一句用户能凭直觉判断的**产品语言**（如「安装 = 备好 还是 装即用」），交用户拍板「进步 / 退步、是否接受」。

## 产出风格样例（节选自 rust-sdk#102）

**舞台**：桌面 App 的 Marketplace。plugin 带两类能力——技能（Skills Tab 可见可点）、MCP 工具（挂上后 Agent 才能调）。按钮：安装 / 启用 / 停用 / 卸载。

**现状（半亮）**：点「安装」→ Skills Tab「唰」冒出技能（已可用），但配套 MCP 工具没出现；卡片还杵着「启用」。用户 OS：「我只点了安装，怎么技能自己亮了？而且一半能用一半不能用？」→ **安装偷偷开了一半的灯，两个按钮职责在用户心里错乱。**

**改后（全亮）**：点「安装」→ 卡片「已安装·未启用」，技能与工具都不出现，像手机装了 App 摆桌面上没打开。点「启用」→ 技能与 MCP 工具**一起点亮**，一个总开关啪一下全屋亮。

**跨场景权衡**：两段式对「想先囤着 / 看配置再用」的用户是进步；但对「点了就是要用」的用户多了一步，可能是退步 → 引出「装 App vs 装浏览器插件」的产品语言选择，交用户定。

## 与既有门控的关系

- 协议门控管「协议合规」，本门控管「用户体感」——**正交，都要过**。
- 本门控在**动代码 / 定协议前**，把「用户觉得是退步」的风险前置消化。
