# 内嵌隔离审查硬门控（add-feature / fix-issue 共用）

> **单一源**。`add-feature`（Step 6 实现后）与 `fix-issue`（Step 7 绿灯后、Step 8 收尾前）各在自己的插入点引用本文件，不复制流水线逻辑。

## 目的

Skill 在主会话执行时，上下文掺杂本次实现过程与既往判断，对刚写的代码满意度天然偏高，难以客观发现问题。把审查交给隔离上下文的 `code-reviewer` 子代理（全新 context、不继承父会话历史），从根上消除"实现者偏见"，无需手动新开会话。

## `--review` 参数

分级词表**复用** `/fix-review`，不另立心智（见 `skills/fix-review/SKILL.md` Step 0）。用**带名标志**而非位置参数——这两个 Skill 首参是自由文本描述，位置参数会冲突。

| 值 | 含义 |
|----|------|
| `block` | 仅修 🔴 阻塞项（**默认**，最严硬门控） |
| `all` | 修 🔴 + 🟡 |
| `discuss` | interview 模式逐条讨论架构决策，不直接改码 |
| `none` | 跳过内嵌审查（仅限下方"比例性豁免"场景） |

## 比例性豁免（避免每次小改都跑两轮子代理）

满足任一即可视为琐碎改动，按 `--review none` 跳过门控，并在收尾时**显式说明已豁免**：

- 修 typo / 纯文档 / 注释改动
- 单文件、单行级、无逻辑分支的琐碎改动
- 用户显式指定 `--review none`

其余一律执行下方硬门控流水线。

## 流水线

```
实现 / 修复完成
   │
   ▼
① 拉起 code-reviewer 子代理（隔离上下文、只读）
   · 任务中传入「变更意图」：需求 / 验收标准 / 父 Issue 北极星
   · 不传实现者自评（保无偏）
   · 子代理自行 git diff 取范围 → 产出 🔴/🟡/🟢 报告
   ▼
② 按 --review 等级委派 /fix-review <level>（报告即"当前上下文"来源）
   · block → 只修 🔴   · all → 修 🔴+🟡   · discuss → 逐条讨论不直接改码
   · fix-review 自带"验证问题真实性"，自动过滤子代理误判
   ▼
③ 重拉 code-reviewer 复核 → 仍有 🔴 回 ②，直到 🔴 清零才放行
```

## 硬门控规则

- 存在 🔴 阻塞级问题则本次 Feature / 修复**不算完成**。
- `discuss` 模式下 🔴 仍须有结论才收尾——"结论"可为讨论后记录的架构决策（走 `/fix-review` Step 6 interview），尊重用户显式选择。
- 子代理不能再拉起子代理：审查在隔离上下文、修复回主上下文（主上下文才有 Edit），审查与修复职责天然分离。
