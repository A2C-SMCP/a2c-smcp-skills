# Compare-OSS — a2c-smcp-protocol 专属指南

> 通用流程参见 SKILL.md 主文件。

## 项目上下文

- 形态：MkDocs 协议规范文档仓（`docs/` + `mkdocs.yml`），无运行时代码
- 定位：SMCP 协议规范源头，协议先行体系的根
- 架构约定：见项目根 `CLAUDE.md`

## Phase 0 差异：对比对象形态

对方通常也是**协议/规范**（而非代码库）：spec 文档、schema 定义、官方仓库里的 `specification/` 目录。clone 后重点读规范文本与消息 schema，而非实现代码。

## Phase 1 差异：维度盘点 = 规范条款对照

逐条对照双方规范：消息/事件类型、生命周期（连接/握手/断线语义）、错误码体系、能力协商、扩展机制、版本演进策略。每格标注规范章节号，方便复查。

## Phase 4 差异：无运行时可 bench → 互操作实测

协议仓跑不了 mini-benchmark，质量类线索的证据替代方案（按可信度降序）：

1. **双端参考实现互操作实测**：用 python-sdk / rust-sdk 搭最小双端，构造对方规范声称更优的场景（如某种重连语义、批量消息模式），实测我方协议语义在该场景的行为
2. **对方参考实现行为实测**：跑对方官方 SDK 的对应场景，验证其规范承诺是否兑现（规范写了 ≠ 实现做到了——命名≠强弱闸的协议版）
3. 两者都构造不出 → 结论降级「存疑」，不得升级为「该抄进规范」

**协议特有过滤器**（Phase 5 追加）：对方规范的设计是否绑定其传输假设（如仅 stdio/仅 HTTP）？我方 Socket.IO 语义下该前提是否成立？

## 典型对比对象

- MCP 官方规范（能力/生命周期/传输层对照的第一对象）
- A2A 等 agent 间协议规范（多 agent 协作语义）
- 其他工具聚合/远程工具调用协议
