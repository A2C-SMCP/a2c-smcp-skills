# Compare-OSS — 自研 vs 开源立体对比

把「我方项目/模块」与「某开源项目」在**技术方案 / 选型 / 能力**三维做证据充分的对比，产出**可借鉴线索**或**「我方领先」的可信结论**。

**为何需要专门 skill**：读代码只能看清「对方有什么」，看不清「哪种设计在哪种场景更优」——设计模式的弊端往往要有该方向的长期实践或**实测**才暴露。凭读码就下「该抄」结论几乎必错（`{baseDir}/resources/websearch-guide.md` 中的 AGE 案例即一次真实翻车）。本 skill 的价值是用**分级证据**把对比从「印象」拉到「可信」。

**核心原则：证据分级**（决定每条结论要多少证据）

| 结论类型 | 判断的是 | 证据门槛 |
|---|---|---|
| **事实/清单** | 对方有/没有某能力 | 读码/读规范即可 |
| **选型/工程** | 该用哪个库/引擎/协议 | **当前 Web 实时数据**（stars/发布节奏/版本支持/治理），禁训练印象 |
| **质量/设计模式** | 谁的架构/性能/召回/去重更优 | **构建并运行场景化 mini-benchmark**，禁纯逻辑推演 |

**成功判据**：得到**可借鉴线索**、或**「我方领先」的可信结论**、或**「持平/无可借鉴项」的实测结论**——三者都是成功。不为「有产出」制造「该抄」。但保持「三人行必有我师」的搜寻姿态（多数开源框架确实水，但很少一无是处）。

| 用途 | 工具 |
|---|---|

## Phase 0：对比对象获取与范围界定

### 0.1 对方仓库获取（按参数形态分派）

| 参数形态 | 处理 |
|---|---|
| **本地路径**（目录存在） | 直接使用。若在会话授权目录外，提示用户 `/add-dir` 或改走 clone |
| **Git URL / owner/name** | `git clone --depth 1 <url> /tmp/compare-oss/<name>`；目录已存在则 `git -C ... fetch && git -C ... log -1` 校新后复用 |
| **裸项目名** | 先 WebSearch / `gh search repos <name>` 定位官方仓库，与用户确认后按 URL 处理 |

**锚定分析基线**：`git -C <对方目录> rev-parse HEAD` 记录 commit SHA，写入最终报告。`/tmp` 会被系统周期清理——跨会话续跑发现目录消失时，按 SHA 重新 shallow clone 即可（成本低，结论可复现）。

### 0.2 范围界定

1. 明确**我方模块**（如 python-sdk 的 Computer 端）与**对方项目/模块**。
2. 明确**对比维度**：技术方案 / 选型 / 能力，或用户指定子集。
3. 定位对方核心代码与**测试目录**（后续基准借它的 fixtures）。

**准出**：对比双方、维度、对方仓库本地路径与 commit SHA 已定。

## Phase 1：功能维度盘点（立体对比地基）

**目标**：把双方能力铺成维度表——**全面**是这一步唯一的 KPI。

1. 读双方核心代码/规范，逐维列「能力 × 我方实现 × 对方实现」。
2. 标注每格：双方都有 / 仅我方 / 仅对方 / 实现方式不同。

**准出**：维度对照表，覆盖用户关心的全部维度，无明显盲区。

**反模式**：在这一步就下优劣结论——盘点只回答「有没有/怎么做」，不回答「谁更好」。

---

## Phase 2：候选线索识别 + 证据分级（核心）

对表中每个「对方可能更优」或「我方可能更优」的点，登记为一条**候选线索**，并打 claim 类型标签（见核心原则表）：

- 打**事实**标签 → 已在 Phase 1 解决，直接进报告。
- 打**选型**标签 → 送 Phase 3（Web 核验）。
- 打**质量/设计模式**标签 → 送 Phase 4（基准实测）。**禁止**在此凭逻辑直接判优劣。

**准出**：每条候选线索带类型标签与「所需证据」，无一条「凭感觉」直接进结论。

---

## Phase 3：选型类 → 当前 Web 现状核验

**目标**：产品/选型建议结合**当前**现状，不凭训练期印象。

对每条选型线索，用 WebSearch / `gh api` 拉实时信号并**核实新鲜度**：

| 信号 | 取法 |
|---|---|
| 活跃度/治理 | stars、forks、contributors、最近 commit、release 节奏 |
| 兼容性 | 目标版本支持情况、EOL、破坏性变更 |
| 风险 | 单一维护者 bus factor、停摆史、许可证、厂商锁定 |

**纪律**：任何「X 落后/不行/过时」的断言，先用当前数据复核——训练期印象常已过时。**诚实优于唱赞歌**：同列对方警示与我方短板。方法细节与 `gh api` recipe 见 `{baseDir}/resources/websearch-guide.md`。

**准出**：每条选型线索有当前数据支撑，结论标注数据日期。

---

## Phase 4：质量/设计模式类 → 场景化 mini-benchmark

**目标**：用**实测**而非逻辑判定设计模式优劣。**这是本 skill 最硬的门槛。**

harness 细节按项目见 `{baseDir}/resources/<project>.md`，通用原则：

1. **借各框架自带测试 fixtures 造数据**——省搭建，且天然贴对方的设计假设。
2. **构建数据即构想场景**：造每条数据时写清「它模拟什么真实业务场景」——这一步天然逼你想清对方设计在什么场景成立/失效。
3. **场景真实**：基准跑真实业务负载，不是纯 API 冒烟。
4. **对称**：同数据、同指标，双方各跑自己的最优路径。
5. **跑起来**：产出数字，不是「预计更快」。一次性基准代码放 scratchpad/`/tmp`，不入库。

**协议仓特例**：a2c-smcp-protocol / oasp-protocol 无运行时可 bench——质量类证据改为**双端参考实现互操作实测**（细则见对应 resource），构造不出实验的结论降级为「存疑」。

**准出**：每条质量线索有可复现的实测数字（或明确记录「无法构造可信基准，结论降级为存疑」）。

**反模式**：读码就断言「对方吞吐更高/我方去重更弱」→ 必须落基准；构造不出场景 → 结论不能升级为「该抄」。

---

## Phase 5：结论过滤器

每条「建议借鉴」进报告前，过三道闸：

| 闸 | 检验 |
|---|---|
| **隐藏前提** | 对方该设计绑定的假设在我方是否成立？（对方单进程、无多语言 SDK 对齐义务、托管环境限制等常不迁移） |
| **刻意设计** | 我方现状是否是**刻意取舍**？先找它存在的理由，再谈替换（如协议先行、同步/异步双实现一致性等约束下的设计） |
| **命名≠强弱** | 「有这个能力」不等于「这个能力强」——看起来强的点常经不起实测 |

**准出**：每条借鉴项都通过三闸；未通过的降级为「存疑/不适用」并说明。

---

## Phase 6：报告合成

```markdown
## <我方模块> vs <对方项目> 对比报告
（对方基线 commit <SHA>，数据日期 YYYY-MM-DD）

### 1. 维度对照表（Phase 1）
### 2. 选型对比（Phase 3，带当前数据 + 日期）
### 3. 质量/设计模式对比（Phase 4，带实测数字）
### 4. 结论
- 可借鉴线索：…（每条注明证据来源）
- 我方领先项：…（带实测/数据支撑）
- 存疑/不适用：…（过滤器拦下的，附理由）
### 5. 诚实警示：对方真实风险 + 我方真实短板
```

**成功即产出上述四类之一或多类**——包括「本轮无可借鉴项，我方在 X/Y 维度领先（实测支撑）」这样的领先结论。可借鉴线索若需落地，衔接 `issue-report` / `add-feature` 建 Issue 推进。

## 关键决策点（必须与用户对齐）

| 决策点 | Phase | 问题 |
|---|---|---|
| 对方仓库定位 | 0 | 裸项目名解析出的仓库是否正确？ |
| 对比范围 | 0 | 对比哪个模块、哪些维度？ |
| 证据分级 | 2 | 每条线索是事实/选型/质量类？ |
| 基准可信度 | 4 | 构造的场景是否够真实、够对称？ |
| 借鉴过滤 | 5 | 隐藏前提/刻意设计是否已排除？ |

## 反模式（本 skill 的立身之本）

| 反模式 | 正确做法 |
|---|---|
| 读码就下「该抄/该偷」质量结论 | 质量类必过 Phase 4 场景基准 |
| 凭训练印象说选型「落后/过时」 | 选型类必过 Phase 3 当前 Web 数据 |
| 全面盘点后急着排优劣 | Phase 1 只盘点，优劣走分级证据 |
| 唱赞歌 / 只挑对方亮点 | 同列对方警示与我方短板 |
| 忽略对方设计的隐藏前提 | Phase 5 隐藏前提闸 |
| 把「有能力」当「能力强」 | Phase 5 命名≠强弱闸 |
| 非得找出「可借鉴项」才算成功 | 领先/持平的实测结论同样是成功 |
| 报告不锚定对方 commit SHA | Phase 0 记录 SHA，结论可复现 |
